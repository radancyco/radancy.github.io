The DOM

Designers don't also need to be developers, but it never hurts to have a baseline understanding of how a PSD becomes HTML. While PSDs are strutured with layers, HTML is structured with the DOM.

Document Object Model

  • Websites are ultimately text files (documents)
  • From a structural standpoint, content needs to maintain the same order on all devices - should not "cut and paste" or change module order as screen space expands
  • Having a good content strategy makes it easy to properly order items on a page - know what users want to access first
  • View the source, Luke

Leave the DOM. Enhance the experience.

  • Developers see sites as one long document, but users don't have to
  • Help users avoid thumb fatigue - add helpers for easier navigation
    • Mobile menu buttons (more on that in a moment)
    • "Back to top" jump links
    • Expandable sections (for both mobile and desktop)
  • Do not be fooled by the mythical "fold"
    • Every piece of content on the site does not need to be visible on page load
    • Every page is different - longer pages may be necessary in some cases
    • Use carousels sparingly

@media

  • Media queries translate mobile first design into mobile first development
  • Non-responsive websites have a single stylesheet (CSS) with a single set of rules for how the page should appear
  • Responsive sites still use one stylesheet, but rules are broken into sets by screen size
  • Use mobile as the base, add different styles on as layout requires - site will inherit base styles unless they are overridden by another style in a media query
  • "Breakpoints" come naturally - adjust the layout when the content starts "breaking" the layout
  • Example: four bottom modules on nissan-jobs.tmpqa.com

Getting Around

Navigation Patterns

We are used to seeing navigation at the top of a page, style-wise, but DOM-wise navigation should actually sit at the bottom of the page. We want users to get to the page content as soon as possible, but still be able to easily access the page navigation. With clever design and the right media queries, we can have both.

Two possible patterns:

  1. Off-screen or slide-out menu
    • Accessed on tap of menu button on smaller screens, pushing main content off the screen or sitting on top of main content when open
    • Can be absolutely positioned to stay at the top of the page on larger screens
    • Avoid fixed positioning (always clinging to the top of the screen, as this site has on larger screens - browser support for fixed positioning is not ideal
  2. Jump link menu
    • This site is an example!
    • Menu "button" at the top of the page is just a jump link - throws user down to navigation at the bottom of the screen
    • Include a "back to top" link
    • Creates a cohesive user experience flow on mobile - when users scroll down to the end of the page content, they are presented with the link list to let them to decide where within the page they would like to go next

Just My Type

Arial? Helvetica? How blasé.

  • When web pages are rendered on a user's computer, they can only draw from the font resources on that computer.
  • If that machine doesn't have a font face (such as Helvetica Neue) installed, text that should render in that font will show up in a plain default typeface (such as Arial) instead.
  • To play it safe, some designers stick to the basic fonts found on most operating systems.
  • Others try to use image slices to replace text
    • Accessibility issue (see below)
    • Text in images scaled down for small screens can be almost illegible
  • Typography on the web doesn't have to be boring.

The Wonderful World of Web Fonts

Web fonts allow you to use custom typefaces on a page without having to worry about what fonts are on an individual's computer. Two good sources:

  • Google Web Fonts
    • Free, open-source typefaces
    • No licenses required - use on as many domains as you want
    • Offers both downloads for use in Photoshop and include links for CSS styling on the web
    • Browse by serif, thickness, slant, and width
  • Fonts.com
    • Client must purchase license to use fonts based on average page visits to the site domains on which the fonts will be used
    • Provides access to common copyrighted typefaces, such as variations of Helvetica
  • Font Squirrel
    • Use to convert custom desktop fonts (such as branded fonts made specifically for a client - not for operating system fonts like Helvetica) to web font files
    • Developers can upload files to reference in CSS styles
    • Only use if you have confirmed with the client that they have permission to use this font on the web. Most fonts require separate licenses for desktop publishing and use on the web, and putting a font online without proper license is a violation of copyright law.

A Word of Warning

Every web font loaded on to a page adds additional load on the user's data connection. Loading too many web fonts slows down the user experience, especially on mobile devices. Try to stick to only one or two web fonts on a page.

Accessibility

  • Responsive design needs to respond to more than just screen size
  • Flexibility to accomodate different input types (mouse, keyboard, finger) as well as assistive technologies
  • Principles of accessibility apply to all sites, not just responsive ones
